Hi, I'm **Brian Duan**. You might also find me online as `iceice666` or `justaslime` (`abignoob` when PWNing CTF, since I just a noob).

I'm a student at 🇹🇼 **National Cheng Kung University**,
and I build personal tools, infrastructure, and experiments for the way I want computers to work.

My motto is still pretty simple:

> A lazy guy, only build project when I need.

That sounds like a joke, but it is also fairly accurate.
I usually do not build things to show off.
I build them when the existing tools feel too heavy,
too opaque, too cloud-dependent, too mouse-driven, or simply not shaped for how I think.

This blog is where I stash notes, experiments, writeups, unfinished thoughts,
and occasionally spicy takes before they disappear into some forgotten chat log or random Markdown file.
For the newest notes and articles in order, start at the [feed](/feed).

## What I Care About

Most of my projects orbit around a few recurring themes:

- **Personal computing environments** — tools that make my own workflow feel less broken.
- **Developer experience** — terminals, browsers, editors, navigation, project management, and the glue between them.
- **Local-first AI agents** — not just "AI wrappers", but agents that can fit into real workflows without turning everything into an opaque black box.
- **Systems and infrastructure** — Linux, NixOS, self-hosting, homelabs, networking, and things that should be boring but rarely are.
- **Programming languages and compilers** — interpreters, IRs, parsing, effects, functional programming, and language implementation.
- **Creative technical projects** — games, Minecraft mods, embedded experiments, and anything else that looks fun enough to justify the rabbit hole.

I like small, composable tools. I like explicit state. I like keyboard-first interfaces. I like systems that can be inspected, repaired, and rebuilt when they inevitably break.

I also use AI heavily as a thinking and implementation assistant, but I care about reviewability, reproducibility, and actually understanding the final system. If the result cannot be debugged, it is not really mine yet.

## How I Work

My preferred workflow is keyboard-first, local-first, and heavily customized.

I spend a lot of time thinking about how tools shape attention. A lot of modern software feels optimized for being impressive in screenshots, not for helping someone stay inside a deep working context. I prefer interfaces that are quiet, fast, scriptable, and composable.

My ideal setup is something like:

* a project-aware terminal
* a browser that behaves more like a programmable research environment
* an editor that understands both code and intent
* local agents that can help without hiding the system
* reproducible machines that can be rebuilt instead of manually blessed

I do not think every tool needs to be minimal. But I do think every tool should be honest about what state it owns, what it changes, and how I can recover when it breaks.
